## 1. v1 회고

### 만든 것

| 영역 | 기능 |
|------|------|
| 소통 | 익명 게시글, 댓글, 리액션 5종 |
| 감정 | AI 감정분석 (Gemini), 13개 감정 체계, 캘린더/트렌드/타임라인 |
| 오늘의 하루 | daily post (감정+활동+한마디), 같은 마음, 어제의 반응, 인사이트 |
| 검색 | 풀텍스트 v2 + 감정 필터 + 하이라이트 |
| 인프라 | 3개 레포 동기화, Sentry, Edge Functions, Realtime |

### v1의 본질

**은둔마을은 게시판이다.** 글을 쓰고, 읽고, 반응하는 것. "오늘의 하루"로 진입 장벽을 낮췄지만 근본적으로 게시판이다.

게시판의 한계: **소통이 비동기적이고 일방적이다.** 글을 올리면 누군가 반응할 수도, 안 할 수도 있다. 반응이 와도 "좋아요" 숫자일 뿐이다. 대화가 아니라 방송이다.

### 은둔자에게 진짜 필요한 것

은둔자가 은둔을 벗어나는 과정:

```
1. 혼자 있음 (현재 상태)
2. "나만 이런 게 아니구나" (인식)
3. 누군가와 연결됨 (경험)
4. 조금씩 말을 꺼냄 (시도)
5. 관계가 생김 (변화)
```

v1은 2단계("나만 이런 게 아니구나")까지 해결한다. "비슷한 마음 N명", "같은 마음의 하루"가 이 역할.

**v2는 3단계("누군가와 연결됨")를 해야 한다.**

## 2. v2 핵심: 연결

### 게시판에서 커뮤니티로

| v1 (게시판) | v2 (커뮤니티) |
|---|---|
| 글 → 반응 (숫자) | 글 → 반응 → **대화** |
| "비슷한 마음 3명" (숫자) | "비슷한 마음의 사람" (**존재**) |
| 모든 사용자가 "익명" | 각자의 **익명 정체성** |
| 나의 글만 추적 | **서로의 여정**을 지켜봄 |

### 익명 정체성 (Anonymous Identity)

현재 모든 사용자가 "익명"이다. display_name은 있지만 게시글마다 랜덤으로 바뀐다. **아무도 누군가를 "알아볼" 수 없다.**

v2 제안: **고정 익명 별칭**.

- 가입 시 자동 부여: "조용한 고양이", "따뜻한 부엉이" 등
- 모든 게시글/댓글에 동일 별칭 표시
- 실명 아님, 프로필 사진 없음 — 완전한 익명 유지
- 하지만 **같은 사람임을 알 수 있음**
- "조용한 고양이가 3일째 하루를 나누고 있네" → 자연스러운 관심

**이것이 v2의 핵심 변화다.** 나머지 기능은 이 위에 쌓인다.

### TogetherAll 모델과의 비교

TogetherAll (NHS/대학 배포): 완전 익명이지만 pseudonymous. 같은 사람을 인식 가능. 24/7 전문 모더레이션.

은둔마을: 동일한 pseudonymous 모델이지만 전문 모더레이터 없이 커뮤니티 자정 + 관리자 운영.

## 3. v2 기능 설계

### 3.1 고정 익명 별칭

**DB 변경**: `user_preferences`에 `display_alias` 컬럼 추가.

```sql
ALTER TABLE user_preferences ADD COLUMN IF NOT EXISTS
  display_alias TEXT;
```

**별칭 생성**: 기존 `ALIAS_ADJECTIVES` × `ALIAS_ANIMALS` 조합 (10×10 = 100가지). 가입 시 자동 부여, 변경 불가 (또는 월 1회 변경).

**적용 범위**: posts.display_name, comments.display_name을 별칭으로 고정. 기존 "익명"이 아니라 "조용한 고양이"가 표시됨.

**기존 데이터**: 이전 게시글은 "익명" 유지. 신규 게시글부터 별칭 적용.

### 3.2 대화 (댓글 강화)

현재 댓글은 flat list. 대화가 이어지지 않음.

**v2**: 댓글에 **답글(reply)** 기능 추가.

```sql
ALTER TABLE comments ADD COLUMN IF NOT EXISTS
  parent_id BIGINT REFERENCES comments(id);
```

1단계 답글만 허용 (중첩 불가). "조용한 고양이"의 댓글에 "따뜻한 부엉이"가 답글 → 짧은 대화.

### 3.3 "비슷한 마음" 강화

현재: "비슷한 마음 3명" (숫자만)

v2: 숫자를 더 구체적으로, 하지만 **개인을 특정하지 않음** (R3 프라이버시 리스크 해결).

```
비슷한 마음
· 이번 주 비슷한 마음 5명
· 오늘 같은 감정의 하루 3개 (같은 마음의 하루 — v1에서 구현 완료)
```

별칭을 노출하지 않음. 집계 데이터만. 개인의 감정 패턴을 동의 없이 다른 사용자에게 보여주지 않는다.

### 3.4 알림 (In-App)

expo-notifications 미설치 상태. Push 알림 전에 **In-App 알림**부터.

**알림 테이블**:

```sql
CREATE TABLE notifications (
  id BIGSERIAL PRIMARY KEY,
  user_id UUID NOT NULL REFERENCES auth.users(id),
  type TEXT NOT NULL, -- 'reaction', 'comment', 'reply'
  post_id BIGINT REFERENCES posts(id),
  actor_alias TEXT, -- "조용한 고양이"
  read BOOLEAN DEFAULT false,
  created_at TIMESTAMPTZ DEFAULT now()
);
```

**알림 유형**:
- "조용한 고양이가 당신의 하루에 공감했어요" (리액션)
- "따뜻한 부엉이가 댓글을 남겼어요" (댓글)
- "조용한 고양이가 답글을 달았어요" (답글)

**UI**: 앱/웹 상단에 알림 벨 아이콘 + 읽지 않은 수 뱃지.

Push 알림은 사용자 수 증가 후 결정. In-App만으로도 "내 글에 반응이 있었다"는 피드백 루프가 만들어짐.

## 8. 리스크 분석 및 해결 설계

### R1. 별칭 충돌 (치명적)

**문제**: 현재 ALIAS_ADJECTIVES(10) × ALIAS_ANIMALS(10) = **100가지**. 사용자 100명이면 충돌 확률 100%. 같은 별칭이 2명 이상에게 부여되면 "같은 사람 인식" 기능이 깨진다.

**해결**: 별칭 풀 확장 + 숫자 접미사.

```
30 형용사 × 30 동물 = 900 기본 조합
충돌 시 → "조용한 고양이 2", "조용한 고양이 3" 자동 부여
```

```sql
-- 별칭 생성 RPC
CREATE OR REPLACE FUNCTION generate_display_alias()
RETURNS TEXT AS $$
DECLARE
  v_adj TEXT; v_animal TEXT; v_alias TEXT; v_count INT;
BEGIN
  v_adj := (SELECT adj FROM unnest(ARRAY['조용한','따뜻한','빛나는','단단한','부드러운','깊은','고요한','맑은','순수한','활기찬','차분한','소박한','온화한','신중한','느긋한','수줍은','평화로운','담백한','은은한','투명한','잔잔한','고운','서늘한','묵직한','가벼운','아련한','선명한','소슬한','포근한','청명한']) AS adj ORDER BY random() LIMIT 1);
  v_animal := (SELECT a FROM unnest(ARRAY['고양이','고래','여우','부엉이','토끼','사슴','하늘소','반딧불','두루미','바람새','수달','곰','거북이','다람쥐','고슴도치','판다','펭귄','올빼미','참새','나비','해파리','달팽이','무당벌레','잠자리','도마뱀','개미','벌새','두더지','카멜레온','해달']) AS a ORDER BY random() LIMIT 1);
  v_alias := v_adj || ' ' || v_animal;

  SELECT COUNT(*) INTO v_count FROM user_preferences WHERE display_alias = v_alias;
  IF v_count > 0 THEN
    v_alias := v_alias || ' ' || (v_count + 1)::TEXT;
  END IF;

  RETURN v_alias;
END;
$$ LANGUAGE plpgsql;
```

**풀 크기**: 30 × 30 = 900 기본 + 숫자 접미사로 사실상 무제한.

**검증**:
- [ ] 1000명 동시 가입 시 충돌 없음
- [ ] 같은 별칭 2명 이상 불가 (UNIQUE 제약)

### R2. 스토킹/추적 (심각)

**문제**: 고정 별칭 = 한 사람의 모든 글/댓글을 추적 가능. "조용한 고양이"를 검색하면 그 사람의 모든 활동이 드러남. **은둔자에게 이것은 공포다.**

**해결**: 별칭은 보이지만 **검색/필터링 불가**.

- 별칭 기반 사용자 검색 기능 제공하지 않음
- "비슷한 사람"(V2-3)에서도 **별칭이 아닌 감정 분포만** 표시
- 글 목록에서 별칭이 보이지만, 별칭을 탭해도 아무 일도 없음 (프로필 페이지 없음)
- search_posts_v2에서 display_name 검색 제외 (이미 FTS 대상 아님 — 확인 필요)

**추가 안전장치**: 별칭 변경 기능 (분기 1회). 불안하면 새 별칭으로 전환 가능.

**검증**:
- [ ] 별칭으로 검색해도 결과 안 나옴
- [ ] 별칭 탭해도 프로필/글 목록 안 보임
- [ ] 별칭 변경 후 이전 글의 display_name도 갱신되는지 (또는 유지)

### R3. "비슷한 사람" 프라이버시 (심각)

**문제**: "조용한 고양이 — 이번 주 무기력 3회"를 공개적으로 보여주는 것은 **타인의 감정 패턴을 동의 없이 노출**하는 것.

**해결**: V2-3 재설계 — 별칭 없이 **익명 집계만** 표시.

```
변경 전: "조용한 고양이 — 이번 주 무기력 3회"
변경 후: "이번 주 비슷한 마음 5명" (별칭 없음, 숫자만)
```

개별 사용자의 감정 패턴을 다른 사용자에게 보여주지 않는다. 집계 데이터만 — "비슷한 마음의 사람이 있다"는 사실, 누구인지는 아님.

### R4. 답글 남용/괴롭힘

**문제**: 고정 별칭으로 특정인을 대상으로 반복 답글 가능. "조용한 고양이"의 모든 글에 부정적 답글을 다는 사람이 나올 수 있음.

**해결**:

1. **답글 제한**: 같은 댓글에 답글 1개만 (본인 답글 중복 불가)
2. **soft_delete 확장**: 답글도 기존 soft_delete_comment() RPC로 삭제 가능
3. **차단 기능 (V2 신규)**: 특정 별칭의 글/댓글을 내 화면에서 숨기기

```sql
CREATE TABLE user_blocks (
  blocker_id UUID NOT NULL REFERENCES auth.users(id),
  blocked_alias TEXT NOT NULL,
  created_at TIMESTAMPTZ DEFAULT now(),
  PRIMARY KEY (blocker_id, blocked_alias)
);
```

차단 = 팔로우의 반대. 팔로우는 안 하지만 차단은 필요. **피하고 싶은 사람을 피할 수 있어야 안전**.

**검증**:
- [ ] 차단한 별칭의 글/댓글이 피드에서 숨겨지는지
- [ ] 차단당한 사람은 차단 사실을 모르는지
- [ ] 같은 댓글에 같은 사용자가 답글 2개 이상 불가

### R5. 별칭 변경 시 데이터 정합성

**문제**: 별칭을 변경하면 이전 게시글의 display_name은?

**선택지**:
- A) 이전 글도 새 별칭으로 일괄 변경 → "조용한 고양이"의 글이 갑자기 "따뜻한 부엉이"로 → 읽는 사람 혼란
- B) 이전 글은 이전 별칭 유지, 신규 글만 새 별칭 → 같은 사람인데 다른 별칭 → 연결 끊김
- C) 이전 글도 새 별칭 + "별칭이 변경되었습니다" 표시 → 복잡

**해결**: **B안** — 이전 글은 이전 별칭 유지. 연결이 끊기는 게 오히려 프라이버시 보호. "과거의 나와 현재의 나를 분리할 수 있다"는 것이 은둔자에게 안전감을 줌.

### R6. 알림 테이블 데이터 증가

**문제**: 모든 리액션/댓글에 알림 생성 → 사용자 N × 활동 M → 무한 증가.

**해결**:

1. **30일 자동 삭제**: 읽은 알림은 30일 후 삭제 (cron 또는 cleanup RPC)
2. **동일 게시글 알림 병합**: "조용한 고양이 외 3명이 공감했어요" (개별 알림 아닌 집계)
3. **인덱스**: `(user_id, read, created_at DESC)` — 미읽음 우선 조회

```sql
CREATE INDEX idx_notifications_user_unread
  ON notifications (user_id, created_at DESC)
  WHERE read = false;
```

### R7. 알림 트리거 성능

**문제**: `toggle_reaction()` RPC 내에서 알림 INSERT가 추가되면 리액션 응답 시간 증가.

**해결**: 알림 생성을 **비동기로** 분리.

- 방법 1: DB 트리거 (AFTER INSERT ON user_reactions) — 트랜잭션 내지만 빠름
- 방법 2: Edge Function (Webhook) — 비동기, 실패해도 리액션에 영향 없음
- **권장: 방법 1** (DB 트리거) — 단순하고 Realtime 호환. 알림 INSERT는 1행 INSERT라 성능 영향 미미.

**검증**:
- [ ] 리액션 토글 응답 시간 변화 측정 (트리거 추가 전/후)
- [ ] 알림 INSERT 실패 시 리액션이 롤백되지 않는지

### R8. 기존 사용자 전환

**문제**: v1 → v2 업데이트 시 기존 사용자의 display_alias가 NULL. 기존 게시글은 "익명".

**해결**:

1. **마이그레이션 시 기존 사용자 전원에 별칭 자동 부여**:
```sql
-- 기존 사용자에게 별칭 부여
UPDATE user_preferences
SET display_alias = generate_display_alias()
WHERE display_alias IS NULL;
```

2. **기존 게시글은 "익명" 유지** — 소급 적용 안 함. v2 이후 새 글부터 별칭.

3. **앱 업데이트 후 첫 실행 시 안내**: "이제 나만의 별칭이 생겼어요: 조용한 고양이"

**검증**:
- [ ] 기존 사용자 전원 별칭 부여 확인
- [ ] 기존 게시글 display_name 변경 없음
- [ ] 신규 게시글부터 별칭 적용
