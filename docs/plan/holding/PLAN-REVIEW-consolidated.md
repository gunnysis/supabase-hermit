- ANR 3건은 프로파일링 없이 코드 수정 불가라고 올바르게 판단. 무리한 추측 수정 안 한 것이 맞음

## 3. 미구현 설계 상세 분석

### 3.1 DESIGN-warmth-retention — 오늘의 온기

#### 서비스 정체성 관점 (기획자 시선)

이 설계의 핵심 통찰은 정확하다: **"글쓰기만이 참여가 아니다"**.

은둔 청년 커뮤니티에서 매일 글을 쓰라는 압박은 역효과를 낳는다. 리액션 하나, 댓글 하나로도 "오늘 마을에 머물렀다"고 인정해주는 방향은 서비스 정체성에 **완전히 부합**한다.

그러나 몇 가지 우려가 있다:

**찬성 근거:**
- 기존 `get_my_streak`이 daily post 작성만 카운트 → 글 안 쓰면 스트릭 0. 이는 은둔 유저에게 좌절감
- 리액션/댓글까지 포함하면 "참여"의 정의가 넓어져서 더 많은 유저가 성취감을 느낌
- recharge_tokens(스트릭 방어권)은 "깨진 스트릭 → 앱 삭제" 이탈 경로를 차단

**우려 사항:**

1. **기능 과잉 위험**: 시 게시판 제거(1주 만에 롤백) 교훈이 아직 생생함. 등불 시각화, 밝기 3단계, 환영 메시지 등은 "있으면 좋은" 기능이지 "없으면 안 되는" 기능이 아님
2. **사용자 부재 상태에서 설계**: 현재 활성 사용자가 극소수. 리텐션 시스템은 **리텐 할 사용자가 있을 때** 의미가 있음
3. **구현 비용**: DB 스키마 변경 + RPC 2개 + 앱/웹 UI 변경 + 상수 추가 — 중규모 작업

#### 기술 관점 (개발자 시선)

설계 자체는 깔끔하다:
- `user_preferences` 확장 2컬럼은 가벼움
- 활동 로그 뷰(UNION)는 성능 이슈 가능성 있음 — posts + comments + user_reactions를 UNION하면 인덱스 활용이 어려움. **Materialized View 또는 활동 요약 캐시**가 필요할 수 있음
- `get_my_warmth_status()` RPC는 매 접속 시 호출 → 쿼리 최적화 필수

**기존 코드와의 충돌:**
- `get_my_streak()` RPC가 이미 존재하고 daily post 기준으로 동작. 이 RPC를 확장할지, 새 RPC로 대체할지 결정 필요
- `StreakBadge` 컴포넌트(앱/웹)가 마일스톤 5단계로 구현됨. 등불 UI로 전환하면 기존 UX와 단절
- `DAILY_CONFIG`, `STREAK_MILESTONES` 상수가 이미 존재 → `WARMTH_CONFIG`와의 관계 정리 필요

#### 판정: **개념은 채택, 구현은 기능 동결 해제 후**

- 스트릭 기준을 "daily post만" → "아무 활동이든"으로 확장하는 것은 **RPC 수정 1건**으로 가능. 이것만 먼저 하는 것도 방법
- 등불 시각화, recharge_tokens, 환영 메시지 등은 사용자 확보 후 단계적으로
- **최소 실행안**: `get_my_streak()` 내부에서 posts + comments + user_reactions의 created_at을 함께 체크하도록 수정 (마이그레이션 1건, UI 변경 0건)

### 3.2 Hermit-Support-Function — 고립 청년 지원 시스템

#### 서비스 정체성 관점 (기획자 시선)

이 기능은 은둔마을의 서비스 방향을 **"감정 커뮤니티"에서 "복지 정보 허브"로 확장**하는 것이다.

**찬성 근거:**
- 은둔/고립 청년은 정보 접근성이 낮음 — 존재하는 지원 프로그램을 모르고 있을 가능성 높음
- "심리적 문턱 점수(threshold_score)"와 "따뜻한 코멘트(warm_comment)"는 은둔마을만의 차별점
- 공공 API 데이터는 무료이고 사회적 가치가 명확

**우려 사항:**

1. **서비스 정체성 변질**: 은둔마을은 "감정을 나누는 익명 커뮤니티"로 시작함. 복지 정보 서빙은 완전히 다른 도메인. 하나의 앱에서 두 가지 성격을 동시에 잘하기 어려움
2. **기술 복잡도 폭발**: ETL 파이프라인 + 멀티 어댑터 + Gemini 2단계 파이프라인 + 벡터 검색 + 하이브리드 랭킹 — 이건 **별도 프로젝트 규모**
3. **무료 플랜 제약**: Supabase Free(500MB DB, 2GB bandwidth) + Gemini 무료 tier에서 매일 배치 수집 + LLM 2단계 가공을 돌리면 한도 초과 우려
4. **유지보수 부담**: 공공 API는 스키마 변경, 장애, 인증 방식 변경이 빈번. 3개 소스(청년센터, 복지로, 지자체)의 어댑터를 지속 관리하는 것은 상당한 운영 부담
5. **사용자 부재**: 현재 활성 사용자가 극소수인 상태에서 대규모 인프라를 구축하는 것은 "기능 과잉" 교훈의 반복

#### 기술 관점 (개발자 시선)

설계 품질 자체는 높다. v6까지 다듬어진 만큼 아키텍처가 체계적:
- 멱등성(content_hash), 지역 코드 표준화, 단계별 LLM 파이프라인 등 실무적 고려가 잘 되어 있음
- 하이브리드 랭킹 가중치(FTS 30% + Energy 40% + Recency 20% + Location 10%)는 합리적

그러나 **현실적 제약**이 크다:

| 제약 | 영향 |
|------|------|
| Supabase Free 500MB | support_programs + content_raw + summary_json이 수백 건만 되어도 상당한 용량 |
| Edge Function 500K 호출/월 | 매일 배치 + 검색 API 서빙 합산 시 한도 근접 가능 |
| Gemini 무료 tier | Flash는 넉넉하나, Pro 단계가 있으면 비용 발생 가능 |
| vector(768) | pgvector 활성화 + 인덱스 = DB 용량 급증 |

**기존 시스템과의 충돌:**
- 현재 DB 테이블 10개, RPC 35개. support_programs 추가 시 도메인이 분리되어 관리 복잡도 증가
- Edge Function이 현재 analyze-post 2개만 존재. 여기에 수집기 3개 + 가공 파이프라인까지 추가하면 운영 부담 급증

#### 판정: **장기 비전으로 보존, 현재 실행 불가**

이 기능은 "은둔마을이 성장했을 때"의 방향성으로서 가치가 있다. 하지만 지금 구현하면:
- 무료 플랜 한도 초과 위험
- 유지보수 부담 급증
- 사용자 없이 인프라만 커지는 구조

**대안 제안:**
1. **Phase 0 (최소 검증)**: 수동으로 10~20개 지원 프로그램을 큐레이션하여 정적 페이지로 제공. ETL 없이 가치 검증
2. **Phase 0.5**: 사용자가 실제로 조회하는지 확인 후, 자동화 결정
3. **Phase 1~3**: 사용자 수 + 유료 플랜 전환 후에야 의미 있음

## 6. 책임자로서의 소견

은둔마을은 지금 **"만들기"에서 "쓰이게 하기"로** 무게 중심을 옮겨야 하는 시점이다.

4개 문서를 통틀어 볼 때:
- Sentry bugfix와 Writing experience는 **"쓰이는 서비스"를 위한 기반 다지기**였고, 잘 완료됨
- Warmth retention은 **"쓰이게 된 후"의 유지 전략**으로 방향은 맞지만 시기가 이름
- Support function은 **"서비스 확장"**으로 가치 있지만, 현재 규모에서는 과잉

기능 동결 기간(2026-03-22~)의 원칙을 재확인한다:
> **새 기능 NO. 기존 기능의 안정성과 사용자 확보 YES.**

완료된 2건은 archive로, 미완 2건은 plan/에 보존하되 실행은 보류. 유일하게 즉시 가능한 것은 warmth-retention의 **최소 실행안**(get_my_streak RPC 확장)이며, 이것도 사용자 요청 시에만 진행한다.
